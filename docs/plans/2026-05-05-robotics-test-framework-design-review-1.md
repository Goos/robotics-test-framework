VERDICT: NEEDS_REVISION

## Summary Assessment

The design is conceptually coherent and the crate layout/HAL boundary discipline is genuinely good, but several load-bearing claims in the doc do not survive contact with Rust ownership rules, basic numerical analysis, or the listed v1 scoring goals. In particular: the port ownership story is under-specified (the world cannot drive a producer once the Rx is moved into the controller without some structure the doc never names), the SNN microstep example silently accumulates phase error, the `Stack` goal cannot be evaluated under "released objects stay put", and the `viz` crate's "never imports `arm`" promise is incompatible with where `Visualizable` impls live.

## Critical Issues (must fix)

1. **Port ownership / driving the producer side is unspecified.** The doc says (a) `port<T>` returns `(PortTx<T>, PortRx<T>)`, (b) `world.attach_joint_encoder_sensor(...)` returns the Rx for the controller to own, and (c) "Sim pushes sensor readings reflecting world state at `t` (each sensor only fires on its scheduled tick)." For (c) to be true the *world* must retain the `PortTx` after handing the `PortRx` out. That is fine for a single sensor but the design also says "Sim-side ops execute in stable order (sorted by port id) each tick" — implying a uniform port registry the sim iterates. Nothing in §4/§5 explains: who owns the Tx half, how the registry is keyed, and how a sensor's scheduled callback knows which Tx to push to. Worse, the actuator path is the mirror image: the controller owns `PortTx<JointVelocityCommand>` and the world holds the Rx — but who calls `drain` and how is it tied back to the right joint? Without an explicit `PortId`/registry type in `core` (or `sim`) and a documented "world holds Tx-for-sensors and Rx-for-actuators, indexed by stable id", the central determinism guarantee ("sorted by port id") is undefined. Fix: spell out the registry. Show the type that `attach_*` records on the world side and where the per-tick iteration happens.

2. **`PortKind` is policy but the API exposes mechanism mismatched to the policy — silently.** `PortRx` exposes `try_recv` / `drain` / `latest`. But `latest()` only has well-defined semantics on `LatestWins`; `drain()` only makes sense on `Buffered` (and arguably `Fifo`). The doc explicitly says "Both endpoints see a uniform read/write API" and "PortKind is policy; the API is mechanism" — so a controller calling `latest()` on a `Fifo` port will compile and run. What does it return? The doc doesn't say. Three options, all bad if undocumented: (a) it's a logic error caught at construction-time (then it's not "uniform"), (b) it returns `None`/the head non-destructively (silent semantic mismatch — bug magnet), (c) it returns the most recent enqueued item but doesn't clear (subtle, breaks FIFO invariant). Fix: either make `PortKind` part of the type (`PortRx<T, Fifo>` etc.) so wrong calls don't compile, OR define and document the semantics for every (kind, method) pair, OR collapse the API to a single read method whose behavior is determined by kind. Punting this is exactly the kind of "silently-wrong behavior" the framework is built to prevent.

3. **SNN microstep example accumulates drift.** §8.2's `let n = ((t - self.last_t) / self.inner_dt).round() as usize;` followed by `self.last_t = t;` is wrong whenever `(t - last_t) / inner_dt` is not integral. Example: outer tick `dt = 1ms`, `inner_dt = 0.3ms`. First call: `n = round(1/0.3) = 3`, advances 0.9ms of SNN sim time, but `last_t` is set to the full outer tick. Repeat: the SNN steadily falls behind real-time at 10%/tick. Over a 15-second test (the doc's own example deadline), that is a 1.5s drift — which for an SNN that depends on temporal coding is not a minor numerical issue, it changes outputs. The fix is the standard fixed-step accumulator: track residual time, run `n = floor(accumulator / inner_dt)`, subtract `n * inner_dt` from the accumulator. This is well known; the design should not ship the buggy version as the canonical example, because that example will get copy-pasted into the first SNN controller. (Bonus: `Time(u64)` nanos minus `Time(u64)` nanos divided by an integer-nanos `inner_dt` should be integer division — there's no float involved in the right answer at all.)

4. **Multi-rate sensor scheduling rounding is unspecified and the doc dodges the question.** §5 says "each sensor only fires on its scheduled tick" with sim at 1000 Hz. If a sensor `RateHz(333)` is requested, the period is 3.003 ms — does the framework (a) round to nearest tick (3ms → 333.33 Hz, drifts off the requested rate), (b) reject non-divisible rates at construction (then the API silently rejects perfectly reasonable rates), (c) use a phase accumulator (correct, but you still get jitter at +/- one tick — controllers must tolerate it). The choice has a real impact on the HAL promise (issue 6 below): a controller that depends on "encoder fires every 3 ticks exactly" works in sim and breaks on hardware. Fix: pick (c), document the +/- one tick jitter, and require the camera/encoder reading to carry the *actual* sample timestamp so the controller can't accidentally rely on the schedule.

5. **`Stack` goal is incompatible with "released objects stay put".** §5 (released objects stay put, no dynamics) and §6 (`Stack` is a v1 goal). For `Stack(A, on=B)` to score under kinematic-only release, the controller must release A while A is already in physical contact with B's top — which means the design implicitly requires a controller that can servo to within sub-millimeter of B's surface and release at exactly that instant. That's far harder than "stack" in any normal physical sense, where you release from a small height and gravity does the rest. Either: (a) drop `Stack` from v1 (consistent with "no premature flexibility"), (b) ship a one-off "snap to top of supporting object on release" rule for the stack predicate (a mini-special-case of dynamics), or (c) bring forward the v2 gravity-fall feature flag now. Pretending §6 and §5 are mutually consistent on this point is the design papering over the gap.

6. **The HAL "controller code unchanged" promise is not a real promise without explicit framework constraints on the controller.** §8.3 calls this "the HAL promise made concrete," but the promise only holds if the sim and the real-hardware adapter expose identical *behavioral* contracts on the ports — not just identical types. Real hardware has: jitter on sensor arrival, occasional drops, latency in actuator commands taking effect, encoder noise, clock skew between the controller's clock and the encoder's sampling clock, and so on. A sim-only controller that does `if encoder_rx.latest().unwrap().timestamp == self.last_tick + Duration::from_millis(1) { ... }` will run perfectly in sim and be totally broken on hardware. The `faults` crate (§8.4) is the right idea but it's optional. To make the HAL promise actually load-bearing, the framework needs to either (a) enable a "realistic by default" mode where `attach_*` already injects bounded jitter/drop/latency that the real-hardware adapter is also guaranteed to respect, or (b) document the precise behavioral contract every `PortRx` is allowed to violate, so a controller author has a checklist. Without one of these, "HAL" is a coding convention, not a framework property.

7. **`viz` snapshot collection contradicts the "viz never imports arm" claim.** §3 says `viz → sim (NEVER → arm)`, and §7 says `arm` impls `Visualizable` for the arm. For the snapshot to include arm primitives, *something* must, per tick, walk the world and call `append_primitives` on the arm. Three possibilities: (i) `sim` walks scene-objects-only and the arm is invisible — then the visualizer is useless for the actual robot. (ii) `sim` somehow walks "all `Visualizable` things" — but `sim` doesn't depend on `arm`, so the world (which is `arm::ArmWorld`) must register its visualizables back to `sim` via a trait object — which means `Visualizable` *must* be object-safe (the doc never says it is, and `&mut Vec<(EntityId, Primitive)>` with `&self` is fine, so OK in principle, but the doc should say so explicitly). (iii) snapshot collection lives in `harness` (which depends on both `sim` and the world type) — plausible, but then the doc should say so and `viz` is downstream of `harness`-produced snapshots. The doc as written does not commit to any of these and the dependency arrows do not work for option (iii) without a dependency-arrow change. Fix: pick one and write down where the per-tick `collect_snapshot(t)` actually runs and what it iterates.

8. **`Goal<W: WorldView>` + `RunnableWorld::View` will need a GAT (or a lifetime parameter on `WorldView`) the moment the view borrows from the world.** The signatures shown — `fn evaluate(&self, world: &W) -> Score` with `W: WorldView` — work fine if `W` is the *owned* world. But the doc also says `Goal<W::View>` and `RunnableWorld::View`, implying `View` is a lightweight projection (e.g., `&ArmWorldView<'a>` containing borrowed refs into the world). Then `RunnableWorld::View` either has to be `View<'a>` (requires GATs — fine on modern Rust but worth committing to) or has to be `'static` (then it can't borrow, then it's just the world itself, then `View` is a useless indirection). If `View` is meant to be the world type itself, drop the `View` associated type and write `Goal<W>` directly. If it's meant to be a borrowed projection, commit to GATs and show the signature: `type View<'a>: WorldView where Self: 'a;` and `fn view(&self) -> Self::View<'_>`. Either way, the current sketch is ambiguous and the answer affects what `Goal` impls look like.

9. **Determinism audit is incomplete.** §5's three-line guarantee ("stable order, FIFO, seeded RNG") is necessary but far from sufficient. Real silent sources of nondeterminism this design does not address: (a) `HashMap` iteration order — the entire codebase must use `BTreeMap` or `IndexMap` for any structure iterated during a tick; this is a lint-able invariant the doc should commit to. (b) Floating-point reductions whose order depends on insertion (e.g., summing forces over contacts) — not relevant in v1 because there's no physics, but will become relevant the moment v2 lands; flag now. (c) `f32` transcendentals (`sin`, `cos` for FK) are not bit-identical across libm versions / platforms / `-ffast-math`-equivalents; "bit-identical outcome" across machines is a much stronger claim than "bit-identical on one machine with one toolchain." Either weaken the claim ("bit-identical on a fixed toolchain+target") or commit to a software softfloat / fixed-point integration path. (d) Any use of `rayon` or `std::thread::spawn` in the sim tick is forbidden — say so. (e) `Instant::now()` is forbidden in `core` and `sim` — say so. The current text invites a future bug where someone adds a parallel iteration "for performance" and silently breaks the load-bearing guarantee.

10. **Test isolation / panic handling is missing entirely.** `cargo test` runs tests in parallel threads by default. If a test panics mid-run, do background threads (e.g., a `StreamRecorder` socket writer, a sensor scheduler thread if any) get cleaned up? If `harness::run` ever spawns a thread, this matters — and the doc never says whether `run` is single-threaded. Recommend: explicitly commit to "the entire `run` is single-threaded on the calling thread; no background threads; no global state." Then panic-safety is automatic. If you want a `StreamRecorder` that writes from a background thread, `Drop` it on panic via RAII and document.

## Suggestions (nice to have)

1. **Logging / observability of controller behavior is missing.** When a test fails (`assert!(result.score.0 > 0.9)`), the user gets a number and no trace. A `Recorder` that captures controller-port events (every send/recv with timestamp) — even just behind a feature flag, off by default — would make debugging vastly less painful. The infrastructure already exists (`Recorder` trait); just add a `ControllerEvent` variant alongside `SceneSnapshot`.

2. **Sensor noise model is unmentioned.** §8.4 lists fault combinators but no Gaussian-noise / bias / quantization combinator for sensor values themselves. Add one (it's a 20-line `PortRx<T>` wrapper given a seed) — without it, the determinism+seed story has nothing to actually exercise on the sensor side.

3. **`Score(f64)` is fine for v1 but consider `Score { value: f64, breakdown: SmallVec<(&'static str, f64)> }` from day one.** The marginal cost is negligible; the marginal benefit when debugging is large; the doc itself flags multi-component breakdown as a known v2 need. Adding the field empty doesn't violate "no premature flexibility" — it's a struct field, not a feature.

4. **Crate count: 6 is on the high side but not unreasonable.** `core` + `sim` + `arm` + `harness` is the load-bearing four. `viz` is justified (out-of-process boundary). `faults` is borderline — it could live as `sim::faults` for v1 with no loss, since nothing else depends on it and there's no public API to protect. Consider collapsing `faults` into `sim` until a second consumer materializes (i.e., `arm-real`), which won't happen in v1.

5. **`Sequence([Pick, Place])` composite goal is YAGNI for v1.** The brainstorm says start with `ReachPose`, `PickObject`, `PlaceInBin`. `Sequence` adds a state machine (which goal is active? when do I transition? how do I score a partial sequence?) that is its own design problem. Defer it; you can always add it later as a single-file utility once you have two real users.

6. **Three `PortKind`s vs one is borderline overkill.** v1 with `LatestWins` only would handle: every sensor (the controller wants the freshest reading), every actuator (the sim wants the freshest command). `Fifo`/`Buffered` only become necessary for events (button presses, contact events) which v1 doesn't have. Recommend ship with `LatestWins` only and add the others when a user (i.e., you) hits the wall. This also dodges issue 2 entirely.

7. **`Time(u64)` nanos overflows in ~584 years — fine — but signed subtraction is a footgun.** `Time - Time` should return a `Duration`-like type; an unsigned wrap if the controller ever passes times in the wrong order will be silent. Either return `Result<Duration, _>` or use `i64` nanos. (This will bite you in the SNN code that does `t - self.last_t`.)

8. **`Recorder: Box<dyn Recorder>` in `RunConfig` forces dynamic dispatch on every snapshot — fine for `NullRecorder`'s zero-cost path only if `NullRecorder::record` is `#[inline]` and the optimizer can devirtualize, which it usually cannot through `Box<dyn>`. If snapshot rate matters, generic-over-`R: Recorder` on `run<W, C, G, R>` and have the user pass `NullRecorder` by value. Moves the cost to monomorphization (longer compiles) but eliminates per-tick virtual call.

## Verified Claims (things I confirmed are correct or well-reasoned)

1. **The HAL boundary as a workspace dependency arrow is genuinely a good idea** and Cargo will mechanically enforce it — controllers depending only on `core` + arm-port-types cannot accidentally `use sim::*`. This is the strongest claim in the design and it holds.

2. **Putting port type definitions (`JointEncoderReading`, etc.) in `arm` rather than `sim`** (§8.3) is the right call for the `arm-real` future. `sim` should not know what an encoder is; only what a `PortTx<T>` is. This decision pays off cleanly.

3. **The two-phase setup (build world → wire ports → construct controller → run)** is a natural consequence of "controller owns its endpoints" and works fine in Rust — the world's `attach_*` methods take `&mut self`, return owned `PortRx<T>`/`PortTx<T>`, and the controller takes them by value in `new`. There's no lifetime knot here as long as port endpoints are owned (no borrow back into the world). The harness then takes the world by value. This composes.

4. **`Controller::step(&mut self, t: Time)` with no observation/action arguments** is the right generalization. Gym-style controllers fall out as one configuration; SNNs fall out as another; the framework imposes nothing about cadence or coupling. This is genuinely subsuming the right space.

5. **The `Recorder` boundary keeping tests headless by default** is correct and well-scoped. Default `NullRecorder`, opt-in to `RerunRecorder` via env var or config — exactly right for `cargo test` ergonomics.

6. **"Goals can see the world; controllers cannot"** is the right asymmetry and the trait surfaces enforce it (Goal takes `&W`, Controller takes nothing). This is a clean expression of test-vs-system-under-test.

7. **Out-of-scope list (§9) is unusually disciplined** for a personal project and the right things are deferred: physics, torques, meshes, multi-robot, real hardware, data-defined scenarios. This is the design's biggest virtue — most personal-project designs sneak half of these in.

8. **Implementation order (§10) is correct** — `core` → `sim` → `arm` → `harness` → first goal → first end-to-end test before `viz` and `faults`. Steps 1-6 really are the minimum coherent slice; 7 and 8 really are independent. No notes.
